Guide UX Lab

A code-first prototyping initiative that replaced click-through mockups with a working product artifact for planning, GTM alignment, and engineering handoff.

Code-First Design AI-Assisted Workflow Platform Prototyping

Executive Summary

Context

Guide's launch required alignment across product, design, engineering, and GTM teams simultaneously. Traditional handoff artifacts were ineffective for validating behavior-level decisions under pressure, creating ambiguity that slowed implementation.

Role

Senior Product Designer leading a code-first design practice. I authored interaction-critical flows and state logic directly in TypeScript/Next.js, while facilitating collaborative review across engineering, product, and GTM partners.

Challenge

Click-through mockups communicated layout convincingly but left interaction logic, state transitions, loading states, and edge cases open to engineering interpretation, often surfacing misalignment late in the development cycle.

Objective

Create a working prototype that made interaction behavior, component APIs, and state management explicit, allowing design intent and engineering constraints to be reviewed together before implementation.

Outcome

Reduced handoff ambiguity, improved cross-functional confidence in scope, and established a repeatable code-first design pattern that future PDLC work now uses as standard practice.

Problem Scope

Design handoff traditionally separates concerns: designers produce visual layouts and interaction flows, while engineers interpret that documentation and implement behavior. Under typical timelines this works, but with compressed planning cycles and complex state logic, translation gaps increase risk.

Traditional click-through mockups excel at communicating visual hierarchy and layout. But they struggle with:

  • State transitions and edge cases that emerge only during interaction
  • Loading states, error states, and permission-based UI variations
  • System rules and data relationships that affect component behavior
  • Navigation logic and context preservation across workflows

Questions about these details surfaced during implementation, creating rework pressure and delaying engineering decisions.

Project Constraints

Compressed timeline

Product definition and engineering execution ran in parallel. Planning could not afford iteration cycles between design and engineering artifacts.

Complex interaction requirements

Guide's search, filtering, and detail workflows required nuanced state management and conditional rendering that static prototypes could not adequately represent.

Multi-stakeholder alignment

Product, engineering, GTM, and design teams needed shared understanding of behavior before resource allocation and messaging decisions were locked.

Reusability requirement

The approach had to establish patterns applicable beyond one release, supporting future platform decisions and reducing handoff friction as standard practice.

Production-aligned code

The prototype stack had to match implementation technologies (TypeScript, Next.js) so engineering could directly reference patterns instead of rewriting from mockups.

Designer autonomy with AI support

I led design and code authorship personally, using AI to accelerate iteration and test coverage rather than relying on engineering to prototype design concepts.

Design Strategy

The Lab shifted from artifact-to-artifact handoff to outcome-focused collaboration. By authoring interaction logic directly in code, we made design assumptions explicit and could review them against engineering constraints in real time, not sequentially.

01

Interaction-first architecture

State logic and component APIs were designed before visual refinement. This ensured behavior was sound and implementable before designers invested in detailed styling or animations.

02

Runnable artifact as shared reference

Product, engineering, and GTM teams reviewed the same prototype. Questions about behavior could be tested directly instead of debated against static mockups.

Component API clarity

Code-First Workflow Architecture

The Lab organized design work around a production-ready tech stack. Components were authored as TypeScript/React modules with explicit state management, making interaction models testable and reviewable without separate documentation.

Design-in-code iteration cycle

I authored features directly in VS Code alongside AI assistance. Product and engineering accessed the working prototype via a shared development environment, allowing real-time feedback on interaction behavior rather than markup-to-artifact translation.

This reduced interpretation gaps and allowed scope discussions to center on actual behavior constraints instead of estimated effort from mockups.

Pattern library and handoff clarity

Reusable component patterns were authored with explicit variants for all supported states (loading, error, permission-gated, empty state). This made the mapping from design decisions to implementation checklist items clear.

Engineering could reference the prototype directly during implementation rather than translating between design documentation and code requirements.

Key Patterns & Decisions

Direct code authorship

Interaction flows and state logic were authored in TypeScript/React, not in separate design tools. This ensured design intent was preserved in implementation-ready code.

Engineering reviewed behavior directly instead of translating from mockups, reducing interpretation drift.

Explicit state modeling

Component state was declared upfront with TypeScript interfaces. This made data relationships, permission logic, and conditional rendering visible before implementation.

Product and engineering could discuss state shapes and transitions as concrete models instead of abstract concepts.

AI-assisted iteration

AI support accelerated scenario expansion and variant coverage. I could explore edge cases and error states faster, moving discovery earlier and reducing implementation surprises.

AI also helped scale test coverage and documentation, maintaining prototype quality under compressed timelines.

Shared development environment

Product, engineering, and GTM accessed the prototype via a live dev server. Questions could be answered immediately by running scenarios instead of debating abstract descriptions.

This transformed feedback cycles from async review to real-time collaboration, improving alignment speed.

Variant-first component design

Components were authored with all supported states concurrently (loading, error, success, empty). This made conditional rendering explicit and ensured no edge cases were overlooked.

Engineering inherited a complete state model instead of discovering state requirements during implementation.

Documentation in code

Component usage, prop interfaces, and interaction rules were documented as code comments and TypeScript types. This reduced the need for separate documentation artifacts.

Engineers could learn component contracts by reading the prototype directly, treating it as canonical reference.

Collaboration

The Lab was designed to improve cross-functional collaboration across a compressed timeline. Product, engineering, and GTM teams accessed the same working prototype, allowing discussions to center on actual behavior and real constraints rather than estimated effort or abstract requirements.

Weekly reviews included live walkthroughs of interaction patterns, state transitions, and edge cases. This format made misalignment visible immediately and resolvable in the moment.

Engineering used the prototype as the implementation reference, reducing translation effort and allowing stricter alignment between design intent and shipped behavior.

GTM partners reviewed the prototype to understand user-facing behavior and messaging positioning, improving launch readiness and customer communication.

The shared artifact accelerated planning discussions and reduced ambiguity around scope and resource allocation because everyone reviewed the same behavior specification.

Reflection

Code-first design is not about designers writing production code. It is about designing behavior in the same language that will implement it, eliminating translation layers and ambiguity.

The Lab's success came from treating the prototype as a collaboration artifact, not a design exercise in isolation. Early engineering involvement shaped component APIs and state models based on implementation realities, not theoretical ideals.

AI assistance proved valuable not for replacing design judgment but for expanding coverage—discovering edge cases, populating variant states, and accelerating iteration when timelines were compressed.

The largest opportunity from code-first design is moving behavior discovery left. Questions about state transitions, edge cases, and system rules surfaced during prototype review instead of implementation, when changes are cheaper and faster.

This practice scales beyond prototyping. The same approach—shared artifact, explicit state models, multi-stakeholder review—improves collaboration for any complex product decision, regardless of timeline pressure.
